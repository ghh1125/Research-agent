# AI 投研初筛研究系统 PRD

版本：Demo 可验收版  
面向读者：产品经理、投资研究团队负责人、分析师、非技术业务同学  
最后更新：2026-04-22

---

## 0. 一句话说明

本系统不是“告诉你买什么”的 AI，也不是普通聊天机器人。它是一个面向投研团队的 AI 研究操作系统，帮助用户把一句模糊的研究想法，快速变成有来源、有证据、有风险、有缺口、有下一步动作的初步研究报告。

用户输入：

> 宁德时代当前是否值得进入深度研究？

系统输出：

- 研究对象是谁，是否上市，属于什么市场
- 应该从哪些角度研究
- 找到了哪些官方、专业、普通来源
- 哪些证据支持研究判断，哪些证据构成反证
- 财务、现金流、行业、治理、估值等变量是否有足够证据
- 当前判断可信度如何
- 是否建议进入深研、加入观察池，还是暂缓研究
- 下一步应该补什么数据、用什么关键词搜、优先找哪些来源
- 一份可以追溯到证据的标准化研究报告

# 一、产品概述

## 1.1 产品背景

在信息过载与研究资源有限的环境下，投资研究团队普遍面临以下问题：

- 研究线索过多：每天大量公司、行业、政策、事件信息涌入，难以快速判断哪些值得投入时间。
- 研究效率低：分析师需要手动搜索资料、整理数据、提炼结论，重复劳动高。
- 证据链分散：新闻、财报、公告、行业报告分散在多个渠道，难以统一验证。
- 判断不透明：很多结论依赖个人经验，缺乏结构化推理过程与可追溯依据。
- 研究协同弱：分析师、风控、投资经理之间信息同步成本高，讨论效率低。

当前具体痛点如下：

| 痛点 | 当前具体表现 |
|---|---|
| 研究起点模糊 | 分析师用一句话描述研究意图，没有标准化的研究对象和目标定义流程 |
| 来源质量参差 | 搜索到的资料混杂官方公告、自媒体文章、导航页面，缺乏分层筛选 |
| 证据无法追溯 | 结论难以回溯到具体来源，复盘时无法确认判断依据 |
| 缺口无法显式记录 | 研究员知道“还缺什么”，但没有结构化方式记录和传递 |
| 报告可读但不可验证 | 现有报告工具生成的文本无法直接对应来源，读者只能盲信 |

随着大模型、检索、结构化推理、PDF 解析和自动化工作流技术成熟，AI 已经具备辅助完成“研究初筛、证据整理、判断建议、报告生成”的能力。

本系统的核心思路是：

> 不直接让 AI 给答案，而是让 AI 先走一遍标准研究流程，再给出有证据约束的初步判断。

---

## 1.2 产品目标

打造一个面向投研场景的 AI Research Operating System，帮助用户从模糊研究问题出发，快速形成有证据支撑的初步研究判断，并给出下一步研究动作建议。

核心目标：

- 降低研究启动门槛
- 提高初筛效率
- 缩短资料搜集时间
- 提升研究过程可解释性
- 沉淀标准化研究流程
- 帮助研究团队判断“什么值得继续研究”

本阶段目标不是替代正式投研决策，而是解决投研工作的第一步：

> 这件事值不值得继续花时间研究？如果值得，下一步应该查什么？

---

## 1.3 目标用户

当前版本主要面向买方机构分析师，也适用于以下内部研究场景：

| 用户类型 | 使用目的 |
|---|---|
| 买方分析师 | 快速启动一个新标的、新行业、新事件研究 |
| 基金经理 | 判断一个线索是否值得分配研究资源 |
| 研究主管 | 检查研究框架、证据质量和缺口是否充分 |
| 风控人员 | 快速发现红旗信号、监管风险和证据薄弱点 |
| 产业资本研究人员 | 快速摸底潜在投资对象、产业链机会和风险 |
| 家办或自营研究人员 | 建立观察池，形成可复盘的初筛记录 |

当前 Demo 默认按“单人研究工作台”设计，暂不包含团队权限、审批流和多用户协同。

## 1.4 产品核心价值

### 1. 从问题到判断的一站式研究流程

用户输入一个问题，例如：

- 拼多多增长是否可持续？
- 宁德时代当前是否值得进入深度研究？
- 腾讯当前估值是否还有研究价值？
- 贸易企业违约原因是什么？
- 某经营权模式有没有合规风险？

系统自动完成：

```text
研究对象识别
→ 研究问题拆解
→ 金融快照获取
→ 官方与专业资料检索
→ PDF 与网页内容解析
→ 有效证据提取
→ 关键变量归一化
→ 覆盖度判断
→ 多角色推理
→ 初步研究判断
→ 下一步研究动作
→ 标准化报告
```

### 2. 证据驱动，而非聊天驱动

系统不会直接凭空生成观点。所有结论都尽量绑定来源与证据。

输出包括：

- 来源网址
- 来源类型
- 来源可信度
- 证据片段
- 支持观点
- 反向证据
- 证据缺口
- 证据评分

### 3. 帮助判断研究资源优先级

系统会输出研究处理建议，例如：

| 内部状态 | 面向用户表达 | 含义 |
|---|---|---|
| `deep_dive_candidate` | 建议进入深度研究 | 当前证据足以支持继续投入研究时间 |
| `watchlist` | 加入观察池 | 有研究价值，但证据还不够强，需要继续跟踪 |
| `deprioritize` | 暂缓研究 | 当前证据不足或风险较大，暂不建议投入更多时间 |

注意：这些是研究流程建议，不是买入、卖出或持仓建议。

### 4. 模拟真实投研团队协作

系统内置多角色视角：

| 角色 | 作用 |
|---|---|
| 资料员 | 整理公开事实、数据、公告和证据 |
| 风控官 | 主动寻找红旗、处罚、现金流、负债和治理风险 |
| 反方分析师 | 挑战主线判断，寻找反证和风险缓解证据 |
| 主研究员 | 综合证据、变量、风险和反证形成初步判断 |
| 投资经理 | 把研究判断转成研究流程建议，例如深研、观察、暂缓 |

这样用户看到的不只是“一个答案”，而是一组类似投研小组讨论后的结构化结果。

### 5. 自动生成可验证研究报告

最终输出标准化报告，包括：

- 研究问题
- 研究框架
- 来源结构
- 多角色视角
- 关键变量
- 核心发现
- 主要风险
- 反方逻辑
- 催化剂与触发点
- 结论压力测试
- 不确定性与证据缺口
- 初步判断
- 投资层判断
- 下一步研究建议
- 证据引用

---

## 1.5 产品定位

本系统不是告诉用户买什么的 AI，而是帮助用户更快判断：

- 什么值得研究？
- 为什么值得研究？
- 当前证据够不够？
- 还缺什么证据？
- 下一步该怎么研究？

| 适用场景 | 不适用场景 |
|---|---|
| 投研团队内部初筛 | 自动交易和仓位管理 |
| 新题材、新标的快速摸底 | 面向外部客户的投顾建议 |
| 分析师辅助产出研究草稿 | 替代人工尽调和正式投资决策 |
| 研究复盘时的证据回溯 | 全量财务数据库与复杂估值引擎 |
| 研究经理审查证据链质量 | 多用户权限与审批系统 |

# 三、核心概念解释

为了让非技术读者理解，以下是系统里几个重要词的解释。

## 3.1 Topic：研究主题

系统把用户输入的一句话变成一个结构化研究主题。

用户输入：

> 华为股票研究价值

系统会识别为：

- 研究对象：华为
- 对象类型：非上市公司
- 上市状态：未公开上市
- 研究目标：不是直接做股票估值，而是研究经营质量、产业链机会和潜在上市情景

## 3.2 Question：研究问题

系统会把一个大问题拆成 5 到 8 个研究问题。

现在每个问题有两个表达：

| 字段 | 给谁用 | 例子 |
|---|---|---|
| `content` | 给分析师看 | 量化过去 8 个季度 Services 业务收入占比提升对自由现金流转化率的影响 |
| `search_query` | 给搜索引擎用 | Apple Services revenue gross margin operating cash flow |

这样可以避免“分析师问题太细，搜索引擎搜不到”的问题。

## 3.3 Source：来源

来源是系统找到的资料，例如：

- 公司官网
- 年报 PDF
- 交易所公告
- SEC 文件
- 专业财经媒体
- 券商研报
- 普通网页

系统会给来源分层：

| 层级 | 含义 | 例子 |
|---|---|---|
| 官方来源 | 公司、交易所、监管机构发布 | SEC、HKEX、巨潮资讯、公司 IR |
| 专业来源 | 专业财经、数据源、研究机构 | Bloomberg、Reuters、券商研报、数据 provider |
| 普通内容来源 | 普通网页、社区、自媒体 | 社区讨论、博客、聚合转载 |

## 3.4 Evidence：证据

证据是从来源中抽取出的可引用片段。

例如：

> 公司 2025 年营业收入同比增长 18%，经营现金流净额同比增长 12%。

系统会记录：

- 来自哪个 source
- 支持哪个 question
- 是数据、事实、观点还是风险信号
- 是否噪声
- 是否截断
- 证据评分
- 是否支持主判断或构成反证

## 3.5 Variable：关键变量

系统把证据归纳成投研变量，例如：

- 收入增长
- 盈利能力
- 现金流质量
- 负债压力
- 行业竞争
- 治理合规
- 估值锚点

变量不是随便总结的，必须绑定对应证据。比如“盈利能力”至少需要毛利率、净利率、净利润、ROE 等相关指标证据。

## 3.6 Coverage Gate：覆盖度判断

Coverage Gate 用来判断某个研究问题是否真的被证据覆盖。

状态包括：

| 状态 | 含义 |
|---|---|
| 已覆盖 | 证据足够支持这个问题的初步判断 |
| 部分覆盖 | 有一些证据，但不够完整 |
| 未覆盖 | 基本没有有效证据 |

例如，财务问题不能因为抓到一句“公司增长稳健”就算覆盖。至少需要收入、利润、毛利率、现金流等关键指标之一。

## 3.7 Judgment：初步判断

Judgment 是系统基于证据生成的结构化判断，包括：

- 当前结论
- 支撑证据
- 反向证据
- 风险点
- 不确定性
- 证据缺口
- 置信度
- 下一步研究动作

## 3.8 Report：研究报告

Report 是最终给用户看的 Markdown 报告。报告不是单纯润色文本，而是由结构化证据、变量和判断渲染出来的。

# 四、用户使用流程

## 4.1 用户输入

用户在 Streamlit 页面或 API 中输入一个研究问题。

示例：

```text
研究英伟达是否值得进一步研究
```

## 4.2 系统处理流程

```text
1. Define：识别研究对象和研究目标
2. Decompose：拆解研究问题
3. Financial Snapshot：拉取金融快照
4. Official Source Injection：补充官方来源
5. Retrieve：搜索资料
6. Source Governance：来源分层与打分
7. PDF Parse：解析 PDF
8. Extract：抽取证据
9. Variable Engine：归纳关键变量
10. Coverage Gate：判断证据是否覆盖问题
11. Reason：生成初步判断
12. Auto Research：自动补证
13. Investment Layer：生成研究处理建议
14. Role Synthesis：生成多角色视角
15. Report：生成研究报告
```

## 4.3 用户获得结果

用户最终看到：

- 一屏摘要
- 金融快照状态
- 研究问题列表
- 来源结构
- 证据列表
- 关键变量
- 判断与风险
- 自动补证记录
- 投资处理建议
- 可下载 Markdown 报告

# 五、核心功能需求

## FR-001 研究主题定义

### 目标

把用户一句话输入转成清晰研究任务。

### 输入示例

```text
腾讯当前是否值得研究？
```

### 输出内容

- 研究对象：腾讯
- 对象类型：上市公司
- 上市状态：已上市
- 市场类型：港股
- 研究主题：腾讯研究价值
- 研究目标：判断其经营质量、风险暴露和后续研究优先级

### 当前能力

系统支持识别：

- 上市公司
- 非上市公司
- 行业主题
- 信用主体
- 宏观主题
- 事件
- 概念主题
- 基金 ETF
- 商品

---

## FR-002 研究问题拆解

### 目标

将一个大问题拆成可执行的研究问题。

### 当前策略

系统默认使用 LLM-first。

也就是说：

1. 优先相信强 LLM 的理解能力。
2. LLM 输出必须符合结构化格式。
3. 如果 LLM 输出太空泛、数量不足或格式错误，系统自动回退到规则模板。

### 输出要求

每个问题包含：

| 字段 | 说明 |
|---|---|
| `content` | 给分析师看的精细研究问题 |
| `search_query` | 给搜索引擎用的宽泛检索关键词 |
| `priority` | 优先级，1 最高，3 较低 |
| `framework_type` | 研究框架类型，如 financial、industry、risk |

### 示例

```json
{
  "content": "量化过去 8 个季度 Services 业务收入占比提升对自由现金流转化率的影响",
  "search_query": "Apple Services revenue gross margin operating cash flow",
  "priority": 1,
  "framework_type": "financial"
}
```

### 设计原因

分析师问题可以很细，但搜索引擎需要关键词。如果直接把很长的分析师问题拿去搜索，可能没有任何网页标题匹配，导致检索失败。

---

## FR-003 金融快照

### 目标

对上市公司尽量获取基础金融数据，例如价格、市值、估值、利润率、收入增长等。

### 支持对象

- A 股
- 港股
- 美股
- ADR
- ETF
- 加密资产，后续扩展

### Provider 路由

系统会根据市场选择不同 provider。

示例：

| 市场 | Provider 尝试顺序 |
|---|---|
| A 股 | yfinance、akshare、eastmoney 等 |
| 港股 | yfinance、akshare、investing fallback |
| 美股 | yfinance、polygon/massive、finnhub、search fallback |

### 状态要求

金融快照不能只返回 `fetch_failed`。系统需要结构化说明失败原因。

当前支持的状态包括：

- `SUCCESS`
- `PARTIAL_SUCCESS`
- `SYMBOL_NOT_FOUND`
- `UNSUPPORTED_MARKET`
- `PROVIDER_TIMEOUT`
- `PROVIDER_RATE_LIMIT`
- `PROVIDER_DOWN`
- `EMPTY_RESPONSE`
- `PARSE_ERROR`
- `AUTH_REQUIRED`
- `INVALID_SYMBOL_FORMAT`
- `NETWORK_ERROR`
- `FALLBACK_USED`
- `ALL_PROVIDERS_FAILED`
- `UNKNOWN_ERROR`

### 用户可见解释

系统不应该只显示：

```text
Status: fetch_failed
```

而应该显示：

```text
金融快照：未获取成功
原因：provider 对该市场支持不稳定
已尝试备用源：akshare
建议：稍后重试或补充官方财报
```

## FR-004 信息检索

### 目标

搜索并筛选可用于研究的资料。

### 当前配置

| 配置 | 当前默认值 | 含义 |
|---|---:|---|
| Tavily 单次返回 | 8 条 | 每个 query 最多取 8 个搜索结果 |
| 最终保留 source | 15 条 | 最终进入研究流程的来源最多 15 个 |
| 每个问题保留 | 4 条 | fact 类型每个 query 最多收 4 条 |
| risk/counter 保留 | 2 条 | 风险和反证类 query 每个最多收 2 条 |
| 自动补证 | 3 条 | 自动补证每个 query 最多收 3 条 |

### 主检索与定向检索分离

系统将 query 分成两类：

| 类型 | 作用 | 示例 |
|---|---|---|
| 主检索 query | 保证召回，优先执行 | Apple revenue gross margin financial results |
| 定向 query | 定位官方来源，失败不阻断 | Apple site:sec.gov 10-K annual report |

这样做的原因是：在部分网络环境中，`site:` 定向搜索可能返回空或超时。如果把它放在最前面，可能导致检索链路体验变差。

当前策略是：

1. 先跑主检索，确保有基础资料。
2. 再尝试 `site:` 官方定向检索。
3. `site:` 失败不会影响主流程。

## FR-005 来源治理

### 目标

区分来源质量，避免把自媒体、转载、导航页面和官方公告混为一谈。

### 来源层级

| 层级 | 说明 | 示例 |
|---|---|---|
| 官方来源 | 监管、交易所、公司 IR、公司公告 | SEC、HKEX、巨潮资讯、公司年报 |
| 专业来源 | 财经数据、券商、专业媒体 | Reuters、Bloomberg、券商研报 |
| 普通内容来源 | 普通网页、社区、聚合转载 | 博客、社区讨论、自媒体 |

### 官方来源识别信号

系统会综合判断：

- URL 域名
- URL 路径
- 文件名
- 页面标题
- 内容关键词

例如：

- `catl.com/uploads/...annual_report.pdf`
- `sec.gov/Archives/...10-k...`
- `hkexnews.hk/...annual report...`
- `cninfo.com.cn/...年度报告...`

这些会被识别为官方或监管来源。

## FR-006 PDF 解析

### 目标

不要只下载 PDF，而是尽量从 PDF 中提取真正有研究价值的页面。

### 当前策略

系统会优先识别以下内容：

- 主要会计数据
- 营业收入
- 净利润
- 经营活动现金流
- 资产负债表
- 利润表
- 现金流量表
- 毛利率
- 研发投入
- 产能利用率

系统会尽量过滤：

- 封面
- 目录
- 页码
- 释义
- 董事长致辞
- 联系方式
- 公司口号
- 法律声明

## FR-007 证据抽取

### 目标

从来源中抽取可引用、可复核、可打分的证据。

### 证据类型

| 类型 | 含义 |
|---|---|
| fact | 明确事实或事件 |
| data | 含数字、比例、金额、增长率、指标 |
| claim | 来源中的观点或判断 |
| risk_signal | 风险信号，如处罚、下滑、现金流恶化 |

### 证据质量控制

系统会过滤或降权：

- 网页导航
- 广告
- 登录注册提示
- 版权声明
- 目录
- 页码
- 乱码
- 截断数字
- 空泛宣传

例如下面这种内容不能作为主结论证据：

```text
营业收入 84 净利润 67 毛利率 2
```

因为它很可能是表格抽取失败后的残片。

## FR-008 变量引擎

### 目标

把零散证据归纳成关键投研变量。

### 当前变量

| 变量 | 需要的证据 |
|---|---|
| 收入增长 | 营业收入、同比增速、季度收入、分业务收入 |
| 盈利能力 | 毛利率、净利率、净利润、ROE、ROIC |
| 现金流质量 | 经营现金流、自由现金流、资本开支 |
| 负债压力 | 资产负债率、有息负债、短债、流动比率 |
| 行业竞争 | 市场份额、同行排名、客户结构、技术路线 |
| 治理合规 | 治理、内控、监管、处罚、许可 |
| 估值锚点 | PE、PB、EV/EBITDA、同行估值 |

### 方向判断

变量方向包括：

- 改善
- 恶化
- 稳定
- 信号分化
- 方向不明

系统会用规则词典纠正明显错误。例如监管罚款、处罚、调查、诉讼不能被误判为治理改善。

## FR-009 Coverage Gate 覆盖度判断

### 目标

避免“有几句话就算研究完成”。

### 规则示例

| 维度 | 至少需要 |
|---|---|
| financial | 收入、利润、毛利率、现金流、趋势数据之一或组合 |
| credit | 经营现金流、债务结构、现金余额、流动性指标 |
| industry | 市场份额、客户结构、技术路线、同行对比 |
| valuation | PE/PB/EVEBITDA 或同行估值参照 |

如果证据不完整，系统会标记为“部分覆盖”或“未覆盖”。

## FR-010 推理与判断

### 目标

基于证据生成初步判断，而不是直接生成答案。

### 输出包括

- 核心结论
- 支撑证据
- 反向证据
- 风险点
- Bear thesis
- 催化剂
- 证据缺口
- 不确定性
- 置信度

### 置信度

置信度分为：

- 高
- 中
- 低

Demo 阶段系统默认偏保守。缺少官方来源、证据少、冲突大、覆盖不足时，置信度会降低。

---

## FR-011 自动补证

### 目标

当证据不足时，系统不应该马上结束，而是尝试自动补充资料。

### 工作方式

系统会根据证据缺口生成研究动作，例如：

```text
补全年经营现金流、资本开支、自由现金流
```

并生成可执行 query：

```text
{entity} annual report operating cash flow capex free cash flow
```

然后系统会尝试再检索一轮资料。

### 记录内容

自动补证会记录：

- 是否触发
- 选中了哪些动作
- 执行了哪些 query
- 新增了哪些 source
- 新增了哪些 evidence
- 是否有效
- 为什么停止

---

## FR-012 投资处理建议

### 目标

把研究判断转成研究流程建议。

### 输出类型

| 建议 | 含义 |
|---|---|
| 建议进入深度研究 | 当前证据足以支持继续投入研究 |
| 列入观察清单 | 有一定价值，但还要补证或等待触发信号 |
| 暂缓投入研究 | 当前证据不足、风险较高或研究性价比不高 |
| 建立跟踪 | 对主题或事件持续观察 |
| 进入信用复核 | 对信用主体需要进一步风控审查 |

### 重要边界

这不是投资建议，不代表买入、卖出或持仓。

## FR-013 报告生成

### 目标

生成非技术用户可读、且可以回溯证据的报告。

### 报告章节

1. 研究问题
2. 研究框架
3. 来源结构
4. 多角色视角
5. 关键变量
6. 核心发现
7. 主要风险
8. 反方逻辑
9. 结论压力测试
10. 催化剂与触发点
11. 不确定性与证据缺口
12. 初步判断
13. 投资层判断
14. 下一步研究建议
15. 证据引用

### 报告原则

- 不暴露工程字段
- 使用中文业务表达
- 明确证据来源
- 明确置信度边界
- 明确下一步动作

# 七、状态与异常处理

## 7.1 金融快照状态

系统必须区分失败原因。

| 状态 | 用户理解 |
|---|---|
| SUCCESS | 成功获取金融快照 |
| PARTIAL_SUCCESS | 获取了部分数据 |
| FALLBACK_USED | 主 provider 失败，使用备用 provider 成功 |
| SYMBOL_NOT_FOUND | 没找到证券代码 |
| UNSUPPORTED_MARKET | provider 不支持该市场或标的 |
| PROVIDER_TIMEOUT | provider 超时 |
| PROVIDER_RATE_LIMIT | provider 限流 |
| PROVIDER_DOWN | provider 不可用 |
| EMPTY_RESPONSE | provider 返回空数据 |
| PARSE_ERROR | 数据解析失败 |
| AUTH_REQUIRED | 缺少 API key 或权限 |
| INVALID_SYMBOL_FORMAT | symbol 格式不符合 provider 要求 |
| NETWORK_ERROR | 网络错误 |
| ALL_PROVIDERS_FAILED | 所有 provider 都失败 |
| UNKNOWN_ERROR | 未知错误 |

## 7.2 早停策略

系统不应过早停止。

当前策略：

- 如果没有任何来源，且金融快照也不可用，才硬早停。
- 如果来源少但仍有数据，继续抽取证据。
- 如果证据不足，先尝试自动补证。
- 自动补证后仍不足，再生成低置信度研究不足报告。

# 八、非功能需求

## 8.1 可解释性

系统所有结论应尽量绑定证据。报告中要能看到证据引用。

## 8.2 稳定性

外部 provider 失败时，系统不应整体崩溃。应记录失败状态并尝试 fallback。

## 8.3 可追溯性

每个 source、evidence、judgment 都要保留 id，便于复盘。

## 8.4 安全边界

系统不得输出：

- 明确买卖建议
- 仓位建议
- 保证收益
- 替代法律意见
- 未经证据支持的强结论

## 8.5 性能目标

Demo 阶段建议：

| 指标 | 目标 |
|---|---|
| 普通研究问题完成时间 | 1 到 3 分钟内 |
| 来源保留数量 | 默认 15 个以内 |
| 单次搜索结果 | 默认 8 条以内 |
| 自动补证轮次 | 默认 1 轮 |

# 九、验收标准

## 9.1 基础流程验收

输入：

```text
研究英伟达是否值得进一步研究
```

系统应输出：

- 研究对象：英伟达
- 对象类型：上市公司
- 市场类型：美股
- 研究问题：5 到 8 个
- 每个问题有 `content` 和 `search_query`
- 金融快照有 provider attempts
- 来源数不应只有 1 到 2 个，默认最多保留 15 个
- 证据可以追溯到 source
- 报告包含风险、证据缺口、投资处理建议

## 9.2 证据质量验收

系统应做到：

- 官方 PDF 能被识别为官方来源
- 页码、目录、联系方式不进入主证据
- 截断数字不进入主结论
- 财务变量优先使用结构化指标
- 缺少关键指标时 Coverage 不能显示完全覆盖

## 9.3 早停验收

系统不应因为 `sources < 2` 立刻停止。

只有在以下情况下硬早停：

- 没有任何有效来源
- 金融快照也不可用

其他情况应继续抽取证据或自动补证。

## 9.4 用户可读性验收

报告中不应出现以下工程字段：

- `watchlist_entry`
- `deep_dive_candidate`
- `coverage_level`
- `weak_source_only=True`
- `source score=`

应显示为：

- 观察清单
- 建议进入深度研究
- 部分覆盖或已覆盖
- 是否仅依赖弱来源：是/否
- 来源分

# 十、典型用户故事

## 用户故事 1：买方分析师快速判断新标的

作为买方分析师，用户输入：

```text
研究宁德时代是否值得进一步研究
```

系统帮助他：

1. 确认宁德时代是 A 股上市公司。
2. 自动拆出财务、现金流、产能、技术路线、价格战、海外政策等问题。
3. 检索公司公告、年报、专业来源和金融快照。
4. 提取收入、毛利率、现金流、市场份额等证据。
5. 输出是否值得继续深研，以及还缺哪些证据。

## 用户故事 2：基金经理判断研究资源分配

作为基金经理，用户想知道一个线索是否值得分配分析师。

系统输出：

- 当前置信度
- 主要风险
- 反方逻辑
- 是否建议 deep dive
- 下一步要补哪些数据

基金经理可以据此决定是否安排正式研究。

## 用户故事 3：研究主管审查证据质量

作为研究主管，用户不只看结论，还要看：

- 官方来源有多少
- 专业来源有多少
- 是否只依赖弱来源
- 哪些问题没有覆盖
- 结论是否引用了真实证据

系统报告支持这种审查。

# 十一、当前限制

当前 Demo 仍有以下限制：

1. 外部搜索依赖网络环境，Tavily、SEC、HKEX、巨潮等来源可能受网络影响。
2. PDF 解析依赖文件质量，扫描件或复杂表格可能抽取不足。
3. 金融快照不是完整金融数据库，不能替代 Wind、Bloomberg、FactSet 等专业终端。
4. 估值模块目前只是初筛，不包含完整 DCF 或情景模型。
5. 自动补证当前默认 1 轮，复杂问题仍需人工继续研究。
6. 报告是研究辅助材料，不构成投资建议。
7. 多用户协作、权限、审批、版本管理尚未实现。

# 十四、总结

AI 投研初筛研究系统的核心价值不是“生成一篇看起来专业的报告”，而是建立一条可复盘的研究工作流。

它把模糊问题变成结构化研究任务，把搜索结果变成可评分来源，把原始材料变成可追溯证据，把证据变成变量和判断，最后生成一个带有风险、缺口和下一步动作的研究报告。

最终目标是帮助投研团队更快回答三个问题：

1. 这个问题值不值得继续研究？
2. 当前判断有什么证据支持？
3. 下一步应该补什么证据？

这就是本系统的产品定位。
