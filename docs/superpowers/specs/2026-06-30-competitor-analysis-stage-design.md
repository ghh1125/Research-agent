# 竞品矩阵分析独立阶段设计

## 背景

当前 Streamlit 页面在竞品选择页提前展示五类尽调材料上传控件。用户点击确认后，系统在一次调用中连续执行竞品矩阵分析、深度尽调、估值分析和综合报告。该交互没有体现需求规定的业务顺序：

`竞品发现 -> 用户选择竞品 -> 竞品矩阵分析报告 -> 补充尽调材料 -> 深度尽调`

## 目标

1. 用户选择竞品后，只运行竞品矩阵分析。
2. 单独展示生成的竞品矩阵分析报告，不提供反馈或重新生成功能。
3. 用户确认进入下一步后，才展示五类尽调材料上传控件。
4. 上传材料并确认后，继续运行深度尽调、估值分析和综合报告。
5. CLI 的一键完整运行行为保持不变。
6. 提升竞品分析 prompt 的证据约束、比较一致性和下游可复用性。

## 方案比较

### 方案 A：只拆分 Streamlit 页面

页面增加中间状态，但仍调用现有 `run_after_competitor_selection()`。该方法会直接执行所有后续节点，无法在竞品报告生成后真正暂停，因此不可采用。

### 方案 B：拆分 pipeline 阶段，页面按阶段调用

新增独立的竞品分析方法和后续分析方法。Streamlit 使用两个方法形成明确暂停点，CLI 继续组合调用两个方法。该方案边界清晰、测试直接，采用此方案。

### 方案 C：引入通用工作流状态机

将所有节点改造成可持久化状态机。扩展性更高，但本次只需新增一个暂停点，引入成本和迁移范围过大，不采用。

## Pipeline 设计

新增两个公开方法：

1. `run_competitor_analysis_step(...) -> CompetitorAnalysis`
   - 输入：开始节点输入、项目基本概况、行业深度分析、竞品发现结果、用户选择的竞品 ID。
   - 行为：校验并记录选择结果，仅运行竞品矩阵分析；未选择竞品时返回明确的跳过报告。

2. `run_after_competitor_analysis(...) -> tuple[DueDiligenceBundle, ValuationAnalysis, FinalInvestmentReport]`
   - 输入：全部上游结果、已经生成的竞品矩阵报告和五类尽调文件。
   - 行为：依次运行深度尽调、估值分析和综合报告，不重复生成竞品报告。

保留 `run_after_competitor_selection(...)` 作为兼容包装器，内部组合上述两个方法，避免 CLI 和已有调用方发生破坏性变更。

## 页面状态流

Streamlit 状态调整为：

`form -> review_overview -> review_industry -> select_competitors -> show_competitor_report -> upload_due_diligence -> done`

### 竞品选择页

- 只显示候选竞品和复选框。
- 不显示任何尽调材料上传控件。
- 点击“生成竞品矩阵分析报告”后，仅调用 `run_competitor_analysis_step()`。

### 竞品报告页

- 展示完整竞品矩阵分析 Markdown。
- 提供 Markdown 和 DOCX 下载。
- 不提供反馈、编辑或重新生成功能。
- 只提供“进入深度尽调”按钮。

### 尽调材料页

- 点击“进入深度尽调”后才显示五类材料上传控件。
- 点击“开始深度尽调”后调用 `run_after_competitor_analysis()`。
- 后续生成深度尽调、估值和综合报告。

## 竞品分析 Prompt 设计

Prompt 必须同时使用：

- 目标公司的公司名称、主营业务、产品/服务、落地场景和核心价值。
- 行业报告中的竞争格局、机会、壁垒及目标公司映射。
- 用户最终选中的竞品清单、竞品关系和逐家检索证据。

分析规则：

1. 只分析用户选中的竞品，不得增加、替换或遗漏。
2. 目标公司和所有竞品必须使用统一比较口径。
3. 固定覆盖产品能力、商业模式、客户与场景、技术壁垒、商业化/融资进展五个维度。
4. 竞争优势和劣势必须是相对于目标公司的比较结论，不写无参照的宣传语。
5. 明确区分已验证事实、基于证据的推断和资料不足；无依据时使用“未公开/资料不足”。
6. 不得将融资规模、公司知名度直接等同于产品或技术优势。
7. SWOT 必须同时结合内部能力与外部行业环境，避免与竞品画像简单重复。
8. `positioning_judgment` 必须形成下游可复用摘要，包含目标定位、主要差异化、关键短板、竞争风险，以及对业务尽调和估值可比性的影响。
9. `meta` 记录关键假设、信息缺口和证据置信度。

报告仍保持需求规定的五部分：竞争格局总览、主要竞争对手、能力对比矩阵、SWOT、目标公司定位判断。

## 错误处理

- 竞品分析失败时停留在竞品选择页，保留用户勾选状态。
- 深度尽调失败时停留在材料上传页，不重新运行竞品分析。
- 未选择竞品时生成明确的跳过报告，仍允许用户进入深度尽调。
- 重复点击不会重复添加同名报告下载组件。

## 测试

1. Pipeline 单元测试验证竞品分析方法只触发竞品分析，不触发任何尽调 schema。
2. Pipeline 单元测试验证后续方法复用已有竞品报告，不再次触发竞品分析。
3. 集成测试验证 CLI 兼容包装器仍可完成全部节点。
4. Prompt 测试验证行业竞争格局、目标公司场景、选中竞品和证据约束进入模型请求。
5. 页面逻辑通过状态转换测试或 Streamlit AppTest 验证：
   - 竞品选择页不存在尽调上传控件。
   - 竞品报告生成后先进入报告展示状态。
   - 点击进入深度尽调后才出现五类上传控件。
