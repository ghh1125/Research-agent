# 后续规划：当前实现 vs. 架构图的差距

这份文档对比当前已交付、已测试的版本（严格按 7 节点规格表实现）和你提供的另一张架构图之间的差距，作为后续迭代的参考，**当前版本不因此改动**。

按你的要求，只看"在现有 7 节点 step-based 架构基础上还能加什么"，不涉及换架构（自主 Agent / MCP 那套不在范围内），也不涉及多空辩论估值机制。

## 当前版本的技术路线（已实现、已测试）

固定顺序的 7 节点 pipeline：每个节点该调什么 LLM prompt、该搜什么、传什么输入都是写在代码里的，唯一的人工干预点是竞品确认。没有对接专门的财务/舆情/专利/投融资数据库，证据来源是通用搜索引擎（Serper/Tavily/Google CSE）+ 用户上传文件解析。

## 在现有架构基础上，架构图里提出但当前版本没有的东西

### 1. 深度尽调和估值分析之间的"汇总"节点

当前已经有 `build_due_diligence_bundle()`（[src/nodes/due_diligence/__init__.py](src/nodes/due_diligence/__init__.py)）把 5 份尽调报告汇总成 `risk_register`（按严重度排序）+ `evidence_index`（来源汇总），这部分其实已经覆盖了图中"汇总"节点的大半意图，**基本不算缺口**。如果图里这个节点还有别的意思（比如要生成一份独立的"汇总数据表"产物，而不只是内部传递的结构），需要你确认具体要汇总成什么样的产物，再决定要不要加。

### 2. 对接真实数据库

图中提到的财务数据库、舆情数据库、研报数据库、专利数据库、投融资数据库，当前版本完全没有——证据来源只有通用搜索引擎结果和用户上传文件。这个事情可以直接嵌进现有架构：在 `src/search.py` 的 provider 链路里加一个新的 provider（跟 Serper/Tavily 并列），或者在某个具体节点（比如团队尽调的背调、技术尽调的专利核查）里多调一次专门的数据库 API，不需要改动整体流程结构。

落地前提：每接一个数据库都是独立的数据源集成工作（API 申请、字段映射、可信度评估），工作量比当前的搜索引擎集成大，需要先确认手上有没有这几个数据库的账号/API 权限，没有权限的话这条做不了。

### 3. deal profile / fact table / ic memo 等产物格式

图左上角提到团队最终要产出 deal profile、source pack、fact table、investment thesis、decision scorecard、ic memo 这些不同形态的产物。当前版本的产物形态是"7 个节点各一份 markdown+docx 报告"，跟这套产物体系不是一一对应的。如果要对齐，需要先搞清楚这几个产物各自的字段构成（目前手上没有这部分的字面规格，没法直接对齐，需要你补充）。

### 4. 更多人工复核点

当前只在竞品确认这一处停下来等人工操作。图里在"项目基本概况"和"行业深度分析"之后也画了"人工复核"/"深度分析和反馈记录"的框，意味着可能想要更多人工检查点。这个在现有架构里加并不难——`BPPipeline` 已经拆成了 `run_intake_through_discovery` / `run_after_competitor_selection` 两段式方法，再加一两个暂停点就是再拆一段，模式是现成的，Streamlit 端加一个新的 `st.session_state.stage` 分支、CLI 端加一次终端确认即可。

## 看不清的部分

架构图右下角"节点 1: 智能拓客"等几个详细说明框里的部分文字（尤其是具体 LLM 能力要求段落）因为截图分辨率/字号原因，有些句子识别不完整，没有把没读清楚的内容硬猜成具体需求写进上面的对比里。如果这几块对你重要，麻烦后续直接发文字稿或更清晰的图。

## 建议的优先级（仅供参考，不代表已经决定要做）

1. **更多人工复核点**——如果你觉得现在"只在竞品确认停一次"不够放心，这个改动最小，在现有两段式架构上直接加，不涉及任何范式调整
2. **对接真实数据库**——收益高但成本也高，建议先确认有没有现成 API 权限再排期，权限没有的话先不用考虑
3. **产物格式对齐（deal profile/fact table/ic memo）**——需要你先补充这几个产物的具体字段规格，否则没法评估改动量
